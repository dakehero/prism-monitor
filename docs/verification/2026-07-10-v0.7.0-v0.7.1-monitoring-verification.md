## Architecture Audit

- `MonitoringHost` contains the only `DispatcherTimer` used for monitoring.
- `Win32ProcessSnapshotProvider` contains the only direct `Process.GetProcesses()` call.
- `CompatibilityProcessService`, `IProcessInfoProvider`, and `Win32ProcessInfoProvider` were removed.
- MainWindow has no process timer or provider. Processes, History recording, tray status, Toast detection, and the visible main window consume projections from one `MonitoringSnapshot`.
- Periodic refreshes use the host's in-memory rules/settings configuration. Store reads in MainWindow are limited to explicit History, Filters, and Settings UI operations.
- Coordinator shutdown cancellation is passed through discovery and enrichment; `MonitoringHost.DisposeAsync` cancels and waits for an active cycle.

Audit command:

```powershell
rg -n "new DispatcherTimer|Process\.GetProcesses\(\)|GetRulesAsync|_settingsStore\.GetAsync" src/PrismMonitor.App src/PrismMonitor.Core
```

## Runtime and Render Review

Method: launched the local self-contained ARM64 Release executable and inspected it through Windows Graphics Capture at the default window size.

Observed:

- Processes rendered unique PID rows after initial activation and after manual refresh. A real initial race found during review was fixed with a serialized latest-work gate; the repeated AppleMusic rows did not recur.
- An expanded complete-details row showed aligned PID, CPU time, wrapped executable path, End process, and Ignore controls. No clipping or horizontal scrollbar was present.
- The limited-details line collapses when unused, so it does not leave an empty gap. A naturally limited compatible process was not available during this run; its visible copy and binding remain covered by layout/model tests.
- The vertical process scrollbar, compact architecture badges, long path wrapping, and manual refresh button were visually checked.
- Settings rendered the accurate text: `On battery, background refresh slows to every 30 seconds while the main window is hidden.`
- History rendered 22 summary rows with search/architecture/ignored filters, architecture badges, a vertical scrollbar, and current launch counts.
- Closing MainWindow hid it while the candidate process continued running; a second activation restored the existing window.

Package-identity behavior was out of scope for this render pass, so no MSIX was generated or installed.

An ETW timer/context-switch trace was attempted with `xperf`, but the non-elevated session returned `NT Kernel Logger: Access is denied (0x5)`. No elevation was requested and no trace artifact was produced.

### Remaining Manual Battery Gate

The machine remained connected to AC during this work. Therefore the following roadmap gate is not claimed yet:

- Same-machine real-battery observation of five visible 10-second intervals and five hidden 30-second intervals.
- Runtime timer-expiration/wake count and an interaction-triggered extra capture while on battery.

The exact DC cadence, fallback behavior, coalescing, and provider-invocation accounting are automated, but those tests do not substitute for a real-battery measurement. v0.7.0 should only be marked release-complete after this section is replaced with observed DC results.
