## Architecture Audit

- `MonitoringHost` contains the only `DispatcherTimer` used for monitoring.
- `Win32ProcessSnapshotProvider` contains the only direct `Process.GetProcesses()` call.
- `CompatibilityProcessService`, `IProcessInfoProvider`, and `Win32ProcessInfoProvider` were removed.
- MainWindow has no process timer or provider. Processes, History recording, tray status, Toast detection, and the visible main window consume projections from one `MonitoringSnapshot`.
- Periodic refreshes use the host's in-memory rules/settings configuration. Store reads in MainWindow are limited to explicit History, Filters, and Settings UI operations.
- Coordinator shutdown cancellation is passed through discovery and enrichment; `MonitoringHost.DisposeAsync` cancels and waits for an active cycle.

Audit command:

```powershell
rg -n "new DispatcherTimer|Process\.GetProcesses\(\)|GetRulesAsync|_settingsStore\.GetAsync" src/PrismMonitor.App src/PrismMonitor.Core
```

### Real-Battery Cadence and Wake-Activity Smoke

At 2026-07-10 21:59 +08:00, `GetSystemPowerStatus` reported battery power (`ACLineStatus = 0`) and 79% charge. The candidate remained on battery throughout the cadence measurements.

#### MainWindow Visible Policy

With MainWindow rendered and active for 62.487 seconds, the candidate used 7.625 seconds of CPU time, changed working set from 198,152,192 B to 202,129,408 B, and changed handle count from 1,089 to 1,015. This includes WinUI rendering/composition work and is not used to infer scheduler boundaries.

MainWindow was then minimized without closing it. This leaves `MonitoringHost` in its visible state while removing continuous rendering noise. During a 68.523-second sample:

- CPU delta: 0.21875 seconds.
- Working set: 208,605,184 B to 207,826,944 B (-778,240 B).
- Handles: 1,020 to 1,023.
- Primary activity peaks: 4.03, 14.11, 24.19, 34.27, 44.34, 54.44, and 64.51 seconds.
- Consecutive primary intervals were approximately 10.08, 10.08, 10.08, 10.07, 10.10, and 10.07 seconds.

This covers more than five real 10-second host-visible battery cycles.

#### Hidden-to-Tray Policy and v0.6.1 Comparison

The candidate and a freshly started installed 0.6.1 instance were hidden and sampled concurrently for 166.274 seconds. Battery charge changed from 77% to 76%, and `ACLineStatus` remained 0.

| Build | CPU delta | Working set start | Working set end | Working set delta | Handles start/end | Non-zero 1-second samples |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 0.6.1 installed | 0.015625 s | 127,578,112 B | 127,324,160 B | -253,952 B | 837 / 830 | 1 |
| v0.7 candidate | 0.484375 s | 207,736,832 B | 205,500,416 B | -2,236,416 B | 1,022 / 1,045 | 11 |

The candidate's primary activity peaks were 7.05, 37.28, 67.55, 96.80, 127.01, and 157.21 seconds. Consecutive intervals were approximately 30.23, 30.27, 29.25, 30.21, and 30.20 seconds, proving more than five real hidden-battery fallback cycles. Minor non-periodic samples were also retained in the raw terminal output and are included in the total CPU delta.

The 0.6.1 comparison correctly remained almost idle because its hidden-battery policy was interaction-only. The candidate intentionally performs bounded 30-second fallback work so newly started processes are not missed indefinitely.

#### Immediate Interaction Refresh

After the candidate returned to hidden mode and its immediate visibility-change refresh settled, a 24.201-second sample was started before the next 30-second boundary. The user hovered the candidate's sole tray icon during that window. One CPU activity peak of 0.03125 seconds occurred at 12.09 seconds, well before the next periodic boundary, confirming an immediate interaction-triggered refresh.

The one-second CPU activity samples are a non-elevated wake-activity proxy. Exact coordinator provider counts and request coalescing remain covered by automated `MonitoringCycleDiagnostics` tests. The earlier elevated ETW attempt was not repeated.
