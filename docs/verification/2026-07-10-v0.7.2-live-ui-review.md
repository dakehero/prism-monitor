## Rule Interaction Evidence

A temporary process-name rule for `codex-computer-use` was created and removed after review.

The live sequence verified:

1. `All surfaces` removed matching processes from Processes.
2. History retained the launch summary and marked it `Ignored` in its default `All` filter.
3. Changing the target to `Processes` restored the unmarked History row while keeping the process suppressed from Processes.
4. Changing the target to `Toast` restored the process row.
5. The persisted target value in `app-rules.json` matched the selected menu item.

History intentionally preserves ignored launch summaries for audit and exposes `All`, `Ignored`, and `Not ignored` filters. A History-target rule therefore changes the ignored state rather than deleting durable launch records.

## Issue Found During Review

Changing the only rule from `All surfaces` to `Processes` briefly rendered an empty Rules list before the updated row appeared.

The target persisted correctly, so this was a list reconciliation problem rather than rule loss. The existing row key included `Targets`, which made a target edit look like one removal plus one addition to WinUI.

Commit `4b38093` added deterministic rule reconciliation. A rebuilt Release render then changed a non-matching temporary rule from `All surfaces` to `Processes` and captured the window after 120 ms. The same row remained visible in place with its updated target text; the list did not render empty.
