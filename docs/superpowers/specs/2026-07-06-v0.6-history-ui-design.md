# Prism Monitor v0.6 History and UI Design

## Goal

Prism Monitor v0.6 improves the main window into a more polished operational UI, adds a 30-day compatibility-mode launch history, and makes the Filters page visually stable. The release stays lightweight: it uses the existing process detection loop and does not introduce CPU sampling, ETW sessions, WMI watchers, or a database dependency.

## Scope

Included:

- A redesigned Settings page with native-feeling grouped sections.
- A new History page showing compatibility-mode app launch records from the last 30 days.
- Stable Filters page updates with better layout and interaction.
- Append-only JSONL launch events plus a compact summary file.

Excluded from v0.6:

- Long-term audit storage beyond 30 days.
- SQLite or another local TSDB.
- Export/import for history or ignore rules.
- Notification quiet hours and advanced notification controls from the older roadmap section.

## Information Architecture

The main `NavigationView` has four destinations:

- `Processes`: currently running compatibility-mode processes.
- `History`: recent compatibility-mode app launches.
- `Filters`: ignored process names.
- `Settings`: monitoring, notification, permission, and data options.

Settings remains at the bottom of the NavigationView using the native settings item. Filters remains a normal navigation item because ignored apps are a first-class data view, not a settings subsection.

## History Behavior

The app records a launch event when the existing detection loop sees a compatibility-mode process ID that has not already been recorded during the current app session. This intentionally records "first detected by Prism Monitor" rather than a kernel-perfect process creation event.

Each event stores:

- Process name.
- Process architecture.
- Process ID.
- Executable path when available.
- Started time when `Process.StartTime` is readable.
- Detected time.

The History page shows 30-day summary rows:

- Process name.
- Architecture.
- Launch count.
- First seen time.
- Last seen time.
- Last executable path.
- Ignored status.

The table is sorted by last seen descending by default. The page supports filtering by process name text, architecture, and ignored/non-ignored status.

## Storage

History storage lives under `%LOCALAPPDATA%\PrismMonitor`:

- `launch-events.jsonl`: append-only event log, one JSON object per line.
- `launch-summary.json`: compact 30-day aggregate used by the UI.

The app writes one JSONL line per new detected launch. It updates the summary asynchronously after appending. On startup and after writes, the store prunes records older than 30 days. If `launch-events.jsonl` grows beyond 5 MB, the app compacts it in the background by rewriting only events inside the retention window.

Read failures, malformed JSONL lines, and unavailable paths are tolerated. Bad lines are skipped during summary rebuild; the app should not crash because history storage is corrupt.

## Filters Page

The Filters page no longer refreshes as a side effect of the process list timer. It refreshes only when:

- The user opens the Filters page.
- The user adds or removes an ignored app from the Filters page.
- A Toast quick action adds an ignored app.
- A main process row action ignores an app.

Updates use diff-like collection changes instead of clearing and rebuilding the entire list. The text box keeps focus and typed text when unrelated refreshes happen.

## Settings Page

Settings uses grouped, native-feeling sections:

- Monitoring: ARM64EC inclusion and power-aware refresh status.
- Notifications: notification level.
- Permissions: current elevation state and administrator restart.
- Data: history retention summary and history clear action.

The page avoids large explanatory paragraphs. Short secondary text is allowed only where it clarifies consequences, such as notifications being unavailable while elevated.

## Energy and Responsiveness

History recording reuses the existing background snapshot and does not add another polling loop. All file IO runs off the UI thread. Tray hover, right-click menus, and WndProc paths never synchronously wait for history or process enumeration.

On battery while the main window is hidden, the app continues to rely on interaction-triggered refreshes from the power-aware refresh behavior. History therefore records launches discovered by those refreshes, not every launch in real time.

## Testing

Core tests cover:

- JSONL append and summary aggregation.
- 30-day retention pruning.
- Malformed JSONL line tolerance.
- Filter list diff behavior.
- History row sorting and filtering.

App-level verification covers:

- Release build succeeds for `win-arm64`.
- MSIX can be built, installed, and launched.
- The main window remains responsive.
- Filters page does not clear/rebuild during process refresh.
