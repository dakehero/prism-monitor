# Prism Monitor v0.7.0-v0.7.1 Monitoring Pipeline Design

## Goal

Close the remaining v0.7.0 and v0.7.1 architecture gaps by replacing the two pull-based refresh loops with one background monitoring path, separating lightweight discovery from process enrichment, and making limited metadata a cacheable and retryable state.

The result must remain inexpensive enough to run all day, preserve compatibility-mode detection for x86, x64, ARM64EC, and ARM64X processes, and keep every runtime surface on one rule-consistent snapshot.

## Scope

Included:

- One runtime scheduler for Processes, History, tray tooltip/menu, and Toast notifications.
- A testable Core monitoring engine that coalesces refresh requests and publishes one immutable snapshot per successful cycle.
- Lightweight process discovery with stable first-detected and process-instance identity.
- Background, staged process enrichment with a PID/creation-identity cache.
- Cacheable limited-detail results with retry backoff.
- Adaptive 3-second, 10-second, and 30-second refresh intervals.
- In-memory monitoring configuration so periodic refreshes do not reread rule and settings files.
- Failure isolation between History, tray, Toast, and main-window consumers.
- Automated coordinator, cadence, cache, retry, PID-reuse, and architecture tests.
- A local power smoke record comparing v0.6.1 and the completed v0.7 pipeline.

Excluded:

- CPU sampling loops, ETW CPU profiling, or a full task-manager feature set.
- Administrator elevation for normal monitoring.
- A process-event watcher that introduces a second background monitoring mechanism.
- Rules UI and advanced rule editing, which remain v0.7.2/v0.7.3 work.
- Packaging, signing, release automation, or diagnostics-page redesign.
- Kernel-perfect launch auditing.

## Architectural Decisions

### One Runtime Owner, Testable Core Engine

`PrismMonitor.App` owns a single `MonitoringHost`. The host owns runtime cadence, power and visibility transitions, cancellation, UI dispatch, and fan-out to platform surfaces. Every refresh trigger routes through this host.

`PrismMonitor.Core` owns a platform-neutral `MonitoringCoordinator`. The coordinator owns refresh coalescing, process-instance tracking, enrichment-cache orchestration, rule projection, last-good snapshot state, and cycle diagnostics. The coordinator does not own a timer or depend on WinUI, Win32, or a dispatcher.

This split keeps Windows lifetime behavior in the App while allowing the complete refresh engine to be tested by `PrismMonitor.Core.Tests`.

### Platform Ports

Core defines two narrow ports:

- `IProcessSnapshotProvider` captures lightweight raw `ProcessObservation` values.
- `IProcessEnricher` enriches a snapshot to a requested detail level.

`PrismMonitor.App` implements both ports with Windows system and Win32 APIs. Blocking enumeration, handle queries, PE reads, package lookup, and certificate reads run on a worker thread. No provider method performs synchronous process enumeration on the UI thread before returning its task.

### Single Snapshot Feed

A successful coordinator cycle produces one `MonitoringSnapshot` with a monotonically increasing sequence number, capture time, and surface-specific process lists. Processes, History, tray, and Toast all receive projections from that exact object.

The main window no longer owns a process-refresh timer and no longer calls a process provider directly. Showing the window changes the host cadence, requests a full-detail refresh, and then applies snapshots delivered by the host.

## Data Model

### Lightweight Snapshot

The platform provider returns `ProcessObservation` values containing PID, process name, nullable cumulative CPU time, and nullable creation time. The coordinator turns each observation into `ProcessSnapshotInfo`, which contains:

- PID.
- Process name.
- Nullable cumulative CPU time.
- Nullable process creation time.
- Stable `DetectedAt`, assigned by the coordinator when that process instance is first observed.
- `ProcessInstanceKey`.

The coordinator supplies `DetectedAt` and the final instance key so detection time remains stable across refreshes.

### Process Instance Identity

When creation time is readable, `ProcessInstanceKey` is PID plus creation time.

When creation time is unavailable, the coordinator assigns a first-observed token and keeps it only while the PID remains continuously present with the same normalized process name. The key is replaced when:

- The PID disappears from a captured process list.
- The creation time changes.
- The normalized process name changes for a creation-time-less PID.

Metadata for a creation-time-less identity is revalidated at least every 30 seconds. This limits stale PID-only reuse when a process exits and another process reuses the same PID between capture cycles.

The instance key flows into `CompatibilityProcessInfo` and launch-history recording so cache invalidation and PID reuse use the same identity end to end.

### Enrichment Result

An enrichment result separates compatibility classification from metadata completeness:

- Compatibility state: `Compatible`, `Native`, or `Unknown`.
- Detail state: `Complete` or `Limited`.
- Architecture display value, including ARM64EC and ARM64X.
- Executable path.
- Package identity.
- Publisher/signing identity.
- Icon cache key.
- Last enrichment outcome.

Missing metadata remains `null`; it does not convert a known compatibility-mode process into a native process or remove it from the current snapshot.

CPU time remains nullable through the surface model. The main window renders unavailable CPU time as `Unavailable` rather than dropping the process.

## Enrichment Levels

The coordinator requests the least expensive level that satisfies the current consumers:

1. `Classification`: determine compatibility architecture. Executable-path and PE inspection are allowed only when required to preserve ARM64EC/ARM64X handling.
2. `RuleIdentity`: additionally request only identity fields referenced by active rules. A package or publisher rule therefore works while the main window is hidden without forcing unrelated metadata reads.
3. `Full`: request all non-icon identity details when the main window is visible or a user explicitly refreshes it.

Icon extraction is not part of process enumeration or coordinator enrichment. `ProcessIconProvider` consumes the cached icon key only for visible rows and reuses the icon while that key remains unchanged.

## Limited Details and Retry Policy

Per-process enrichment failures do not fail the monitoring cycle.

- A process already classified as compatible remains visible when path, package, publisher, or later queries fail.
- If classification temporarily fails for a previously classified compatible instance, the cache retains the compatible classification and publishes limited details.
- A never-classified process with `Unknown` compatibility stays internal and is retried, but is not published to compatibility-mode surfaces. This prevents protected native system processes from turning Prism Monitor into a noisy task-manager list.
- Native classification is cached for the lifetime of the process instance to avoid repeating expensive work for unchanged native processes.

Limited or unknown results retry with exponential backoff beginning at 30 seconds and capped at 5 minutes. A successful full result resets the failure count. A newly requested higher enrichment level may run immediately even if a lower-level retry is delayed.

The cache stores the last attempt time, next retry time, failure count, outcome, enrichment level, and metadata. Pruning removes entries for process instances that are no longer active.

## Refresh Scheduling

The App host uses one periodic schedule:

- AC power: every 3 seconds.
- Unknown power: every 3 seconds.
- Battery with the main window visible: every 10 seconds.
- Battery hidden to tray: every 30 seconds.

The following request an immediate refresh without adding another loop:

- Tray hover or right-click.
- Opening the main window.
- Manual refresh.
- Power-source change or resume.
- Monitoring-setting or rule change.

Changing power or visibility restarts the single schedule with the new interval. The 30-second hidden-battery fallback is intentionally retained even when there is no user interaction so short-lived processes are not missed indefinitely.

## Refresh Coalescing

Only one discovery/enrichment cycle runs at a time.

Requests received during a running cycle merge into one pending request. The pending request keeps the strongest requested enrichment level and the most urgent reason. After the active cycle completes, the coordinator runs at most one follow-up cycle. This covers simultaneous and slightly staggered consumers; correctness does not depend on a short cache-duration race.

The old two-second `CompatibilityProcessService` cache is removed once all consumers use the coordinator.

## Configuration

Rules and monitoring settings load once when the App host starts. The coordinator receives an immutable in-memory configuration snapshot.

Main-window changes and Toast ignore actions explicitly reload the affected store, replace the in-memory configuration, and request an immediate refresh. Periodic cycles do not reread JSON files.

Legacy ignored-name migration remains handled by the current rule store and is outside this pipeline redesign.

## Failure Handling

### Capture Failure

If the process snapshot provider fails or a cycle is cancelled during shutdown, the coordinator does not publish an empty valid snapshot. It retains the last successful snapshot and records a failed cycle diagnostic.

### Per-Process Failure

A failed snapshot field or enrichment field affects only that process and field. Other processes continue through the cycle. A known compatible process uses its cached limited state where safe.

### Surface Failure

App fan-out isolates consumers:

- History events are appended as one batch per successful cycle.
- Tray status updates independently.
- Toast detection and delivery update independently.
- Main-window dispatch updates independently.

A History file error does not prevent tray or Toast work in the same cycle. Sink failures are best effort and cannot invalidate the coordinator's successful snapshot.

## History Semantics

History continues to mean “first detected by Prism Monitor,” not kernel-perfect creation auditing.

`LaunchHistoryRecorder` tracks process-instance keys rather than an ever-growing PID set. It removes identities that are absent from the current snapshot, records a reused PID with a new creation identity as a new launch, and writes the snapshot's stable `DetectedAt` and available creation time.

Multiple new launches from one monitoring cycle are written through a batch store operation so summary rebuilding occurs once per cycle.

## Lifecycle

The host starts after App activation has initialized stores, tray, Toast, and the main-window lifetime state. It captures an initial baseline before notifications begin.

Closing the main window hides it, changes cadence to the appropriate background interval, and leaves the host running. Exit cancels the host, stops its single schedule, waits for the active cycle to finish or cancel, detaches power events, and then disposes tray and Toast resources.

## Diagnostics and Power Evidence

The coordinator retains in-memory cycle diagnostics required for tests and manual verification:

- Last successful capture time and duration.
- Refresh reason and active interval.
- Provider invocation count.
- Process counts before and after filtering.
- Enrichment cache hits, misses, retries, and prunes.

This is not a new diagnostics page and does not create another timer or log-writing loop. Manual smoke evidence may be written to an ignored local file first; only the final concise verification record is committed.

## Automated Testing

Core tests cover:

- One provider capture fans out to Processes, History, tray, and Toast.
- Concurrent and staggered requests coalesce without duplicate captures.
- A stronger pending full-detail request is not lost.
- Capture failure retains the last-good snapshot.
- Detection time remains stable for one process instance.
- PID reuse and missing-creation fallback invalidate stale enrichment.
- AC, visible-battery, and hidden-battery intervals are 3, 10, and 30 seconds.
- Cache hits do not repeat path, PE, package, publisher, signing, or icon-key work.
- Limited results remain visible for known compatible processes.
- Retry backoff begins at 30 seconds and caps at 5 minutes.
- ARM64EC and ARM64X classification remains intact.
- History records process-instance reuse and batches one cycle's events.
- Rules with partial metadata use the existing predictable evaluator behavior.

App wiring tests verify that `MainWindow` no longer owns a process timer or process provider, and that App runtime fan-out consumes coordinator snapshots rather than initiating separate scans.

## Manual Verification

The completed implementation must pass:

- Core Release tests.
- Full solution Release tests.
- `win-arm64` self-contained Release build.
- A launched local Release build smoke test for Processes, History, tray, and Toast behavior.
- Real render review if the `Unavailable` limited-detail state is visible.
- A same-machine v0.6.1 versus v0.7 hidden-to-tray battery comparison recording CPU delta, wake behavior, working set, configured cadence, provider invocation count, and observed discovery delay.

Generated packages, local logs, certificates, and raw measurement artifacts remain untracked.

## Success Criteria

The design is complete when:

- There is exactly one runtime refresh schedule and one provider capture path.
- Every runtime surface consumes the same successful monitoring snapshot.
- Hidden-battery monitoring performs a 30-second fallback refresh and interaction refreshes remain immediate.
- Main-window rendering never performs process enrichment synchronously.
- Unchanged process instances do not repeat expensive enrichment.
- PID reuse invalidates stale metadata.
- Known compatibility-mode processes remain visible with limited details when optional metadata is inaccessible.
- ARM64EC/ARM64X detection and current rule behavior remain covered.
- Automated, build, runtime, render, and power evidence required by the roadmap is recorded.
